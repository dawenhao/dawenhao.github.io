    # 1. 系统概述

    XTimeline 是一套 自定义时间轴编辑器框架，设计目标是脱离 Unity Timeline 的 PlayableGraph 依赖，提供一套完全 纯数据驱动、可序列化为 JSON 的时间轴系统，用于游戏中的技能编辑、过场剧情、Buff 配置、控制器状态机等需要"时间轴 + 多轨道 + 片段"的场景。

    # 核心设计理念

    理念 说明
    纯数据驱动 Runtime 数据层完全不依赖 Unity Engine（noEngineReferences: true），可独立于 Unity 运行
    编辑器/运行时分离 通过 4 个 Assembly 严格隔离编辑器逻辑与运行时数据
    Bridge 模式 通过桥接接口（Config、Json、Logger、Audio）解耦外部依赖
    泛型轨道模型 基于 XTimelineTrackAsset<T> 泛型模式，一个轨道类型对应一种片段类型
    Attribute 驱动扩展 通过 [MenuName]、[TrackType]、[TrackColor]、[TrackClipType] 等特性声明式注册轨道
    支持多种配置类型 Skill（.clip）、Director（.director）、Buff（.buff）、Story（.story）、Sequence（.seq）、Controller（.xcontroller）、Blueprint（.bp）、AI（.ai）

    # 3.3 ClipAsset（片段基类）

    属性 类型 说明
    Start double 片段起始时间（秒）
    Duration double 片段时长（秒）
    TimeScale double 时间缩放倍率（默认 1.0）
    DisplayName string 显示名称
    ClipCaps ClipCaps 片段内有效时长结束后的延续状态（None/Looping/Hold）
    PostCaps ClipCaps 片段结束后的延续状态
    AssetDuration float 片段的有效内容长度（如动画时长）
    IgnoreTime bool 是否忽略时间

    关键方法：

    方法 说明
    GetEnd() 返回 Start + Duration
    ToLocalTimeUnbound(time) 全局时间转片段本地时间
    FromLocalTimeUnbound(time) 片段本地时间转全局时间
    SetParentTrack(track) 设置父轨道，自动维护轨道的 Clip 列表
    InTime(target) 判断当前 Clip 是否在 target Clip 的时间范围内
    SupportDragClipLength() 是否支持在编辑器中拖拽修改长度（可覆写）

    # 3.6 XTimelineTrackEntityBase（实体轨道）

    实体轨道是子轨道的容器，代表时间轴上的一个"实体"（如角色、特效、摄像机等）。

    # 附录 A：关键设计决策

    # 决策 原因 权衡
    1 Runtime 不依赖 Unity Engine（noEngineReferences: true） 支持服务端验证/独立工具链/跨平台；使用 System.Numerics.Vector2 替代 UnityEngine.Vector2 需要 Bridge 层桥接所有引擎功能
    2 使用 BaseObject 代替 UnityEngine.Object 提供相同的 ID/相等性语义，但不依赖 Unity 生命周期管理；支持 implicit operator bool 静态递增 kBaseId 有线程安全和溢出风险（见缺陷报告）
    3 泛型轨道 XTimelineTrackAsset<T> 强类型的 Clip 列表，避免运行时类型转换和错误 需要维护双重列表（Clips + _cachedClips），增加同步复杂度
    4 Attribute 驱动注册（[MenuName]/[TrackType]/[TrackColor]/[TrackClipType]） 声明式定义轨道关系、颜色、菜单名，减少硬编码；编辑器通过 TypeCache 反射自动发现 编译期无法校验 Attribute 参数正确性
    5 Bridge 模式（XTimelineBridge） 运行时解耦，编辑器和游戏运行时可绑定不同实现（Config/Json/Logger/Audio） 桥接层本身为静态类，全局状态，需注意初始化顺序
    6 partial class 三区域分区 将 Runtime Data / Runtime Function / Editor 逻辑分散到不同 partial 区域，清晰分层 单个类跨多个文件，增加导航成本
    7 配置保存时排序 Clip（SortClips/SortUsedClips） 确保按时间顺序执行，避免创建顺序导致的 Bug 排序缓存（_cacheSorted）无失效机制（见缺陷报告）
    8 RuntimeId 系统（递增 _runtimeUniqueId） 为每个 Clip/Marker 分配递增 ID，支持运行时 O(1) Dictionary 查找 ID 为 int 但 Dictionary key 为 long，类型不一致
    9 AI 按类型平铺序列化（AIConfig 80+ 列表） 避免多态 JSON 序列化的复杂性，System.Text.Json 原生支持 每新增节点类型需在 AIConfig/ParseToConfig/ParseToGraph 三处修改
    10 Controller 模板/覆写系统 复用状态机拓扑结构，仅替换节点关联资源，减少配置工作量 模板展开使用 DeepCopy，运行时内存消耗增加
    11 AI 节点 ID 从 0 递增查找空闲（GetNewNodeID） 简单直观，保证 ID 不重复 O(n²) 复杂度，节点数多时性能差
    12 编辑器 Action 系统基于反射自动发现 零配置注册，新增 Action 只需继承基类 + 添加 Attribute 启动时 Activator.CreateInstance 开销，无法编译期检查
    13 DiscreteTime 高精度时间（Int64 + 1e-12 tick） 消除浮点精度问题，在帧边界判定上比 double 更可靠 转换频繁时有一定计算开销
    14 Dictionary<Type, IList> 用于 Entity 轨道缓存 灵活支持任意数量的轨道类型，无需修改基类 IList 非泛型接口，取出后需要类型转换
